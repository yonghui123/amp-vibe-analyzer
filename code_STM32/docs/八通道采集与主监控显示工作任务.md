## 1. 需求摘要（具象目标）

用户点击主监控 **Start Acq** 后，系统应：

1. **周期性读取** AD7606 八通道同步采样原始值（硬件层可高于 UI 刷新率）。
2. **按通道设置** 中的 `channel_id`、`enabled`、`ch_type` 将每路数据路由到不同处理路径（不是 raw 自带类型）。
3. **通用通道**：所有「已启用 + 类型=通用」的通道 → 校准后的瞬时值 → 通用折线图多条曲线。
4. **振动通道**：仅 **采集设置** 中选定的 `vibration_ch_num` 一路 → 高频缓冲 → 峰峰值 → 振动折线图（+ 已有包络 CSV 参考线）。
5. **报警（延后）**：§2.4 规则 **仅作设计规格**；编码首期 **只预留接口与空方法体**，采集链路可调用但不产生真实报警；**阶段 8** 再补方法体与 GUI 联动。
6. **UI 刷新**：采集任务约 **20ms** 向 GUI 推送一次显示用数据；GUI 任务继续 5ms 调 `lv_timer_handler()`，不在 GUI 任务内读 ADC。

**不在本专项首期编码范围**（文档保留设计，代码后置）：

- **报警方法体**（§2.4、阶段 8）：防抖计时、状态机、弹窗、记录持久化。
- SD 卡 FatFs 驱动与 JSON/CSV 持久化（见 SD 卡接入专项）。
- 历史曲线、报警记录 SQLite、Excel 导出（P3）。
- PRD 5.2 转速带通 IIR 滤波完整实现（首期可用「不滤波」或简单滑动平均）。
- PRD 5.3 包络线在线生成（首期继续使用采集设置加载的 CSV 包络文件）。

**首期编码目标（阶段 0~7）**：Start Acq → 读 8 路 → 校准路由 → 通用多曲线 + 振动 PP 实时显示；报警调用点存在但 **恒为不报警**。

### 2.4 报警：持续时间防抖（设计规格 · **阶段 8 再实现方法体**）

> **编码策略（v1.2）**：本节为 **需求与设计规格**，阶段 0 只落地 **头文件 + 空函数**；`acq_pipeline` / `acq_task` 在阶段 2~3 预留 **`Alarm_ProcessFrame()` 调用点**，方法体内仅 `(void)` 参数并返回「不报警」。**禁止**在阶段 1~7 实现计时与弹窗。

**原则**（阶段 8 实现时遵循）：某一时刻信号 **开始越过阈值** 后，并 **不能立即报警**；必须在阈值之上 **连续维持足够长的时间**，才认定报警成立。偶尔一次采样偏高（毛刺、干扰）应被过滤掉。

这与 PRD 4.1「超限后计时 → 持续时间 ≥ 延迟 → 才触发」一致，实现上按 **采集任务节拍（默认 20ms）累加越限时长**，而不是「越限 1 个采样点即报警」。

#### 2.4.1 状态机（每路/每个判定对象独立）

```
                    ┌─────────────────────────────────────┐
  每 tick 采样值     │  value < threshold ?                │
 ─────────────────► │    是 → 越限累计时间清零，保持正常      │
                    │    否 → 累计时间 += Δt（如 20ms）     │
                    │         累计 < 要求时长 → 触发中(黄)   │
                    │         累计 ≥ 要求时长 → 报警(红)     │
                    └─────────────────────────────────────┘
```

要点：

| 规则 | 说明 |
|------|------|
| **回落清零** | 一旦 `value < threshold`，累计时间 **立即清零**，需重新从 0 开始计 |
| **Δt 来源** | 使用采集任务实际周期（`ACQ_UI_UPDATE_MS`），与 UI 刷新同频 |
| **单次毛刺** | 仅 1 次 tick 越限 → 累计 20ms ≪ 要求时长 → **不报警** |
| **触发中状态** | 已越限但未满时长 → `STATUS_TRIGGERING`（黄灯），便于现场观察 |
| **报警后** | 保持报警直至用户 **Alarm Reset** 或策略定义的自动恢复（首期：仅手动复位） |

#### 2.4.2 参数来源（与现有 UI 字段对应）

| 报警场景 | 比较对象 | 阈值参数 | **持续时长参数** | 配置位置 |
|----------|----------|----------|------------------|----------|
| **一级：通用信号超限** | 每路已启用通用通道的校准值 | `ChannelConfig.params.general.alarm_threshold` | `ChannelConfig.params.general.alarm_delay_ms`（默认 1000ms） | **通道设置** 表格 |
| **二级：振动包络前置触发** | 采集设置选定的 **通用参考通道** `general_ch_num` | `AcqParams.general_threshold` | **`AcqParams.threshold_time_s`**（0.1~10s，UI「阈值时间」） | **采集设置** → 分组三「阈值设置」 |
| **二级：振动包络超限** | 选定振动通道 PP 序列 vs 包络 | 上/下包络 + `envelope_*_pct` | `AcqParams.alarm_threshold_pct`（超限 **时间占比** ≥ 该%，非单次时长） | **采集设置** |

说明：

- 您提到的 **「超过多少时间之后才开始报警」** 在 **采集设置** 中对应 **`阈值时间`（`threshold_time_s`）**，用于 **通用参考通道越限后，延迟多久才进入振动包络判定**。
- **一级通用报警** 的持续时间在 **通道设置** 的 **`alarm_delay_ms`**（PRD 默认 1s）；若产品希望统一只用采集设置里的时间，需在审阅项中确认（当前按 PRD 两处分工实现）。

### 6.2 类型区分逻辑（本阶段核心）

**每次处理不是「对比匹配」，而是固定索引 + 配置字段：**

```c
for (uint8_t phys = 0; phys < 8; phys++) {
    ChannelConfig_t *ch = ChannelData_GetById(phys + 1);
    if (!ch || !ch->enabled) continue;

    float v = calibrated[phys];
    if (ch->ch_type == CH_TYPE_GENERAL) {
        /* 若在 gen_route 中 → 填入 general_vals[series_index] */
    } else if (ch->ch_type == CH_TYPE_VIBRATION) {
        if (ch->channel_id == vib_ch_id) {
            /* 阶段 5：写入振动环形缓冲 */
        }
    }
}
```

**优化**：`gen_route[]` 在 RebuildRoute 时建好，运行时只遍历 `gen_count`（≤4），不必每 tick 扫满 8 路配置表。

**报警**：本阶段末尾仅 `Alarm_ProcessFrame()` 空调用，不读阈值。

### 6.3 功能边界

- 输出 struct，例如 `AcqDisplayFrame_t { float general[MAX_GENERAL]; uint8_t general_count; float vib_pp; ... }`。
- 本阶段 `vib_pp` 可填 0 或暂用瞬时值占位。
